# Splendor Engine Specification for AI Development

This document provides a technical deep-dive into the `csplendor` engine, specifically designed for developers building AI players or Reinforcement Learning (RL) models.

---

## 1. Core Constants & Enumerations

### 1.1 Gem & Bonus Ordering
The internal array representation for gems (size 6) and bonuses (size 5) follows a strict order that differs from the visual representation in some versions but must be adhered to for backend communication:

| Index | Name | Constant | Category |
| :--- | :--- | :--- | :--- |
| 0 | Diamond | `DIAMOND` (White) | Gem/Bonus |
| 1 | Sapphire | `SAPPHIRE` (Blue) | Gem/Bonus |
| 2 | Emerald | `EMERALD` (Green) | Gem/Bonus |
| 3 | Ruby | `RUBY` (Red) | Gem/Bonus |
| 4 | Onyx | `ONYX` (Black) | Gem/Bonus |
| 5 | Gold | `GOLD` (Wildcard) | Gem only |

### 1.2 Action Types
Actions generated by `MoveGenerator` are mapped to the following integer types:

| Value | Constant | Description |
| :--- | :--- | :--- |
| `0` | `TAKE_DIFFERENT` | Take 1 gem each of 3 different colors. |
| `1` | `TAKE_SAME` | Take 2 gems of the same color (if bank >= 4). |
| `2` | `RESERVE_VISIBLE` | Reserve a face-up card + 1 Gold. |
| `3` | `RESERVE_DECK` | Reserve a card from the deck + 1 Gold. |
| `4` | `PURCHASE` | Buy a card using gems/bonuses/gold. |
| `5` | `VISIT_NOBLE` | Manually select a noble when multiple are eligible. |

## 2. Detailed State Machine

### 2.1 The `waiting_noble` State
A critical feature for AI is the state transition for noble visits.
1. After a player applies a standard move (Take/Reserve/Purchase), the engine checks for eligible nobles based on the player's bonuses.
2. **0 Eligible**: Turn ends, board passes to the next player.
3. **1 Eligible**: Noble is automatically assigned, turn ends.
4. **Multiple Eligible**:
   - `board.waiting_noble` is set to `true`.
   - `board.current_player` **remains the same**.
   - `MoveGenerator::generate_all` will **only** return `VISIT_NOBLE` actions.
   - The player must apply a `VISIT_NOBLE` action to choose one noble.
   - Once chosen, `waiting_noble` becomes `false` and the turn ends.

### 2.2 Final Round & Completion
- When any player reaches 15 points, `board.final_round` is set to `true`.
- The game ends after all players have had an equal number of turns (i.e., when the player who went second finishes their turn).
- **Tie-breaking**: Points > Fewest Purchased Cards > Tie.

## 3. Action Encoding for AI Models

Each `Action` object contains:
- `type`: (int) Action category.
- `take[6]`: (int array) Number of gems taken from bank (negative if returning gems).
- `card_id`: (int) ID of the card being purchased or reserved (-1 if none).
- `deck_level`: (int) 0-2 (mapping to Level 1-3) for deck reservation.
- `from_reserved`: (bool) True if purchasing from hand.
- `gold_as[5]`: (int array) **Crucial for Purchase**. Specifies exactly how many Gold tokens are being used as substitutes for White, Blue, Green, Red, and Black respectively.
- `noble_choice`: (int) ID of the noble selected during `VISIT_NOBLE`.

### Purchase Action Generation
Unlike some engines that simply check "if affordable", `csplendor` generates **every possible way to pay** for a card using Gold.
> [!NOTE]
> If a player has 3 Gold tokens and needs 1 Red and 1 Blue, the engine generates actions for (1 Gold as Red, 1 Gold as Blue) and other permutations if they exist. AI should treat these as distinct choices to optimize Gold usage.

## 4. Reinforcement Learning Tips

### 4.1 State Vectorization
When flattening the state for a Neural Network, consider the following blocks:
1. **Player 0 / Player 1 Stat Blocks**: (Gems, Bonuses, Points, Reserved IDs).
2. **Board Bank**: (Current count of 6 colors).
3. **Board Cards**: (Face-up card IDs flattened + deck counts).
4. **Nobles**: (Remaining noble IDs).
5. **Phase Indicators**: `waiting_noble`, `current_player`, `final_round`.

### 4.2 Reward Shaping
- **Sparse**: +1 for win, -1 for loss.
- **Dense**: Small points for:
  - Incrementing Prestige Points.
  - Buying cards with high point-to-cost efficiency.
  - Denying opponent a key color or noble (requires looking 1 step ahead).

## 5. C++ / Python Integration

The Python extension `csplendor` provides the `Game` and `Board` objects directly. Use `game.legal_actions` to get the list of available moves. 

```python
import csplendor

game = csplendor.Game(seed=42)
while not game.is_game_over():
    # board state
    board = game.board
    current_player = board.current_player
    
    # Check for manual noble choice phase
    if board.waiting_noble:
        # Must pick from VISIT_NOBLE actions
        legals = game.legal_actions
    else:
        # Normal turn
        legals = game.legal_actions
        
    # AI logic to pick an action index
    idx = 0 
    game.apply(legals[idx])
```
